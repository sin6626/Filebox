# FileBox MVP 设计文档（不含 AI）

## 1. 背景与目标

FileBox 是一个 Windows 常驻托盘桌面工具，用于自动收集和管理聊天软件接收文件。

本期 MVP 目标：

1. 监听微信/QQ 文件接收目录。
2. 在侧边栏按时间倒序展示新文件。
3. 提供托盘入口与通知能力。
4. 使用 SQLite 保存历史记录。
5. 支持路径配置的双模式：设置页可改、并写回 `config.json`。

明确排除（当前里程碑不做）：

- AI 摘要
- AI 重命名
- AI 分类/打标签
- 任意 OpenAI/API 推理流程

## 2. 范围定义

### 2.1 In Scope

- 多目录监听（微信/QQ）
- 扩展名过滤（读取 `config.json`）
- 新文件入库
- 侧边栏自动展开/收起
- 基础操作：打开文件、打开所在文件夹、从历史中移除
- 系统托盘菜单（显示/隐藏侧边栏、退出）
- 新文件通知（可配置开关）

### 2.2 Out of Scope

- AI 相关全部能力
- 云同步、账号系统
- 文件内容预览/编辑
- 自动移动文件到统一目录
- 删除原文件等高风险操作

## 3. 总体架构

采用单进程模块化架构，减少 MVP 落地复杂度并保留可扩展边界。

```
main.py
  -> config 模块（读写配置）
  -> storage 模块（SQLite）
  -> watcher 模块（watchdog）
  -> sidebar 模块（PyQt5 UI）
  -> tray 模块（pystray）
```

核心原则：

- watcher 不直接操作 UI。
- sidebar 不直接操作文件系统扫描，只通过 storage 读历史。
- 设置修改统一写回 `config.json`，并触发 watcher 平滑重载。

## 4. 模块设计

### 4.1 `config`

职责：加载、校验、保存配置。

关键能力：

- 加载默认值（字段缺失时自动补齐）
- 校验监听路径与扩展名格式
- 设置页变更后写回 `config.json`

### 4.2 `watcher`

职责：监听目录并产生“新文件事件”。

关键能力：

- 监听多个目录
- 过滤扩展名
- 路径规范化（绝对路径）
- 短时间重复事件幂等处理

### 4.3 `storage`

职责：SQLite 持久化与查询。

建议表结构（可微调）：

- `id`（主键）
- `file_name`
- `file_path`（绝对路径）
- `source_type`（wechat/qq）
- `received_at`（时间戳）
- `exists_flag`（是否仍存在）

说明：同名文件全部保留，按完整路径区分。

### 4.4 `sidebar`

职责：展示与交互。

关键能力：

- 默认按 `received_at desc` 展示
- 贴屏幕边缘自动展开
- 失焦自动收起
- 项操作：打开文件、打开所在目录、移除记录

### 4.5 `tray`

职责：托盘图标与快捷菜单。

关键能力：

- 显示/隐藏侧边栏
- 退出应用
- 错误状态提示（监听目录不可用等）

## 5. 关键数据流

### 5.1 启动流

1. 读取配置。
2. 初始化数据库并建表。
3. 启动 watcher。
4. 初始化侧边栏和托盘。

### 5.2 新文件流

1. watcher 捕获创建/移动完成事件。
2. 校验扩展名并规范化路径。
3. 写入 storage。
4. 触发 UI 刷新。
5. 按配置发出系统通知。

### 5.3 配置变更流（方案 C）

1. 用户在设置页增删路径。
2. 配置模块写回 `config.json`。
3. watcher 接收重载信号并平滑切换监听目录。

## 6. 错误处理与稳定性

- 单目录监听失败不导致应用整体退出。
- 对监听异常做有限重试（退避）。
- 文件尚未写完导致打开失败时给出明确提示。
- SQL 一律参数化，写操作用事务。
- UI 与 watcher 使用线程安全通信（信号/队列），禁止跨线程直接改 UI。
- 日志统一用 `logging`，包含模块前缀和关键上下文。

## 7. 非功能要求

- 目标平台：Windows
- Python：3.10+
- 启动可用：`python main.py`
- 较大目录下保持可用，不因短期突发事件崩溃
- 保持模块边界清晰，便于后续扩展

## 8. 测试策略

### 8.1 单元测试优先级

1. `config`：默认值、校验、回写。
2. `storage`：建表、插入、排序查询、移除。
3. `watcher`：过滤、路径规范化、重复事件去抖。

### 8.2 测试实践

- 使用 `pytest` 与 `tmp_path`。
- watcher 重点测事件处理函数，减少依赖真实等待。
- tray/sidebar 使用 mock，避免测试环境 UI 限制。

### 8.3 命令基线

- 全量：`pytest -q`
- 单文件：`pytest tests/test_file_watcher.py -q`
- 单用例：`pytest tests/test_file_watcher.py::test_detects_new_file -q`
- 按关键字：`pytest -k watcher -q`

### M1：基础骨架

- 项目结构搭建

### M2：核心链路

- watcher -> storage -> sidebar 刷新链路可用
- 托盘菜单与通知可用

### M3：设置与稳定性

- 设置页路径管理 + 写回配置

### MVP 验收标准

- 能监听配置路径并捕获新文件
- 侧边栏默认按时间倒序展示

- 三个文件操作可用（打开文件/打开目录/移除记录）
- 托盘显示正常，可退出应用
- 测试与质量命令可运行

## 10. 风险与后续扩展

已知风险：

- 聊天客户端写文件时序差异会导致事件重复或早触发。
- 不同 Windows 环境下托盘/UI 行为可能存在差异。

应对策略：

- 通过幂等和短时间去抖降低重复事件影响。
- 关键平台行为增加集成验证清单。

后续扩展（非本期）：

- 文件搜索与筛选
- 批量操作
- AI 增强能力（仅在后续里程碑重新评审后再启用）
